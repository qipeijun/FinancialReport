# A股投资决策增强 v1：候选股引擎 + 结构化评分 + 报告解释层

## Summary
在现有 `新闻采集 -> 质量筛选 -> AI生成报告/判断卡片 -> 归档` 的链路上，新增一个独立的 `A股推荐评分引擎`，让“股票推荐”从 prompt 生成改为规则层产出、AI 负责解释。

v1 的明确产出是：
- 候选股清单
- 每只股票的 `总分 + 5维子分`
- 推荐等级 `强关注 / 关注 / 观察 / 回避`
- 推荐理由、主要风险、失效条件
- 完整结构化 JSON 写入 metadata / output JSON，供二期回测使用

v1 不改 SQLite 主业务表结构，不把评分结果写回 `news_articles`；评分结果只写归档 metadata 和导出 JSON，降低迁移风险。

## Key Changes

### 1. 候选股来源先收敛，避免“推荐池失控”
候选股池只允许来自两类来源：

1. `新闻直接提及个股`
- 从标题、摘要、正文中抽取 A 股股票代码/公司名
- 只接受能映射到 A 股证券主表的标的
- 每个候选股必须保留来源文章 ID 列表和证据摘要

2. `主题映射个股`
- 仅针对高置信度主题候选组启用
- 主题到股票的映射不交给 AI 临场编造，改为本地配置表维护
- 配置格式建议：`config/theme_stock_map.json`

默认限制：
- 单次报告最多输出 10 只候选股
- 单个主题最多映射 3 只股票
- 没有直接新闻证据、也不在主题映射配置中的股票，禁止进入评分池

### 2. 评分采用“门槛过滤 -> 分层打分 -> 等级映射”
评分顺序固定：

1. 门槛过滤
- 新闻证据不足：降为“观察”或直接剔除
- 无法取得最近 60 交易日行情：不计算技术分，只能“观察”
- 无法取得估值基准：估值分记中性，不允许给“强关注”
- 存在重大负面风险标签（立案、停牌、业绩暴雷、重大减持等）：直接压级

2. 分层打分（100 分制）
- `新闻催化 30`
- `技术面 25`
- `估值面 20`
- `质量/风险 15`
- `市场环境 10`

3. 等级映射
- `80-100` 强关注
- `65-79` 关注
- `50-64` 观察
- `<50` 回避

附加限制：
- 若 `新闻催化 < 15`，最高只能到“观察”
- 若 `质量/风险 < 6`，最高只能到“观察”
- 若任一核心数据缺失导致 `data_completeness < 0.7`，禁止“强关注”

### 3. 五维评分规则固定参数，避免实现漂移

#### 新闻催化（30）
输入：
- 来源等级
- 独立证据数
- 发布时间新鲜度
- 是否为公司经营/订单/财报/政策直接催化
- 与股票关联强度（直接提及 > 行业映射）

建议规则：
- 直接提及且有 2 个以上独立来源：高分
- 仅聚合源或单一转述源：低分
- 48 小时内事件高于旧闻
- 公司级催化高于泛行业叙事

#### 技术面（25）
固定指标与参数：
- `MA20 / MA60`
- `RSI(14)`
- `MACD(12,26,9)`
- `BOLL(20,2)`
- 近 `5/20` 日量比
- 近 `20` 日价格动量

评分原则：
- 趋势、动量、量价共振加分
- 超买不直接判空，但压低上限
- `KDJ` 仅作辅助确认，不单独计入主分，避免与 RSI/MACD 重复

#### 估值面（20）
默认口径：
- `PE(TTM)`
- `PB(LF)`
- 行业中位数对比
- 历史分位对比（建议 3 年窗口，数据不足时退化为 1 年）

分类处理：
- 普通盈利股：PE + PB
- 金融股：PB 为主，PE 弱化
- 周期股：PE 低权重，改看 PB 和分位
- 亏损成长股：不算 PE 分，估值分默认中低并提示“高估值不确定性”

估值原则：
- 低估只加分，不单独构成推荐
- 高估但强催化可保留“关注”，不能直上“强关注”
- 无行业基准时估值分记中性 10 分，并标记数据不完整

#### 质量/风险（15）
纳入：
- 盈利稳定性
- 是否亏损
- 资产负债压力
- 近期重大负面公告/监管事件
- 高波动/高换手异常

v1 若财务数据不足，可先用简化版：
- 是否连续亏损
- 是否存在重大风险新闻
- 是否 ST/停牌/退市风险
- 是否短期异常波动

#### 市场环境（10）
以 A 股日常决策为目标，环境分只做顺风/逆风修正：
- 沪深主要指数趋势
- 候选股所属行业趋势
- 风格偏好（成长/价值、防守/进攻）
- 风险偏好状态

环境分不决定选股，只调节上限和风险提示。

### 4. 数据接口与缓存边界明确
新增 4 个 provider 接口：

- `SecurityMasterProvider`
  - 公司名/代码标准化
  - 行业归属
  - A股市场状态校验

- `PriceHistoryProvider`
  - 最近 60~250 交易日 OHLCV
  - 支持计算 MA/RSI/MACD/BOLL/量比

- `ValuationProvider`
  - PE/PB/PEG
  - 行业中位数
  - 历史分位所需序列或快照

- `RecommendationScorer`
  - 接收候选股和多维特征
  - 输出统一评分结构

v1 数据落地规则：
- 原始新闻继续进 `news_articles`
- 候选股评分结果仅写：
  - `docs/archive/.../metadata/*.json`
  - `--output` 指定的 JSON
- 可新增本地缓存目录如 `data/market_cache/`，但不写回文章表

### 5. 报告展示形态固定，避免和 Judgment Cards 混淆
`markdown-report` 新增一个独立章节：`股票推荐评分`

固定包含两部分：

1. 推荐总表
- 股票代码
- 股票名称
- 总分
- 推荐等级
- 五维子分
- 数据完整度

2. 个股解释卡
- 推荐理由
- 核心催化
- 主要风险
- 失效条件
- 适用时间窗（默认 `5-20 个交易日`）

`judgment-cards` 不直接塞入完整个股打分表，只允许引用：
- 当前最值得跟踪的主题
- 该主题下最多 1-2 个代表标的
- 标注“详细评分见完整报告”

### 7. 验收脚本同步升级
现有 `run_acceptance.py` 只校验双模式产物和质量，需要新增“评分引擎专项验收”。

新增验收项：
- metadata 中存在 `stock_recommendations`
- 每个推荐对象必须有 `symbol/name/grade/total_score/scores/data_completeness`
- 五维子分之和与总分一致或可推导
- 缺失关键行情时，结果必须降级，不能输出“强关注”
- 无新闻证据时，不得出现高等级推荐
- Judgment Cards 与 markdown-report 不互相覆盖，且模式职责不混淆

## Test Plan

1. 单元测试
- 指标计算：`MA20/60`、`RSI14`、`MACD(12,26,9)`、`BOLL(20,2)`
- 评分映射：阈值、压级逻辑、缺失数据降级
- 估值分支：普通股、金融股、周期股、亏损股
- 候选股归因：直接提及、主题映射、非法候选剔除

2. 集成测试
- 强新闻催化 + 趋势配合 + 估值合理：进入“关注/强关注”
- 强新闻催化 + 技术过热 + 高估值：被压到“关注/观察”
- 仅技术形态漂亮、无有效新闻：不得高推荐
- 无法拿到历史行情：只能“观察”
- 无法拿到估值数据：估值记中性，禁止“强关注”

3. 产物测试
- `markdown-report` 必须有“股票推荐评分”章节
- metadata 必须有结构化推荐列表
- `--output` JSON 必须可被后续回测脚本直接消费
- 同一输入重复运行，规则层推荐结果应基本一致

## Assumptions
- v1 只服务 A 股。
- v1 的默认决策时间窗为 `5-20 个交易日`，不做超短线信号。
- v1 不输出仓位建议，只输出等级和解释。
- v1 不把评分结果写入 SQLite 主表，只写归档 metadata 和导出 JSON。
- 二期再做历史回测、权重调优和收益/回撤验证。
- 若估值或历史行情数据源不稳定，优先保证“可解释降级”，不伪造完整评分。
